# Issue Tracker Workflow

How issues flow on `stonematt/lookout`. Optimised for one-maintainer
operation with AI agents doing PRD authoring, triage assistance, and
implementation.

## Label vocabulary

Every triaged issue carries **exactly one category label** and **exactly
one state label**. The maintainer can override at any time.

### Categories

| Label         | Meaning                          |
|---------------|----------------------------------|
| `bug`         | Something is broken              |
| `enhancement` | New feature or improvement       |

### States

| Label              | Meaning                                                |
|--------------------|--------------------------------------------------------|
| `needs-triage`     | Maintainer hasn't evaluated yet                        |
| `needs-info`       | Waiting on reporter for more information               |
| `ready-for-agent`  | Fully specified, an AFK agent can pick it up           |
| `ready-for-human`  | Fully specified, requires human (judgment / external)  |
| `wontfix`          | Won't be actioned                                      |

State machine:

```
(unlabeled new issue)
        ↓
   needs-triage
        ↓
   ┌────┴───────────────────────────────┐
   ↓        ↓               ↓          ↓
needs-info  ready-for-agent  ready-for-human  wontfix → closed
   ↓
   (reporter replies → back to needs-triage)
```

## Lifecycle

### 1. Create

Issues land via:

- **Reporter form** (Web UI / API) — usually starts as a bug report or
  feature request.
- **`/to-prd`** — turns the current conversation into a PRD-shaped issue
  body. Applies `enhancement` + `needs-triage` by default.
- **Direct `gh issue create`** — same defaults as `/to-prd`.

All new issues should land with `needs-triage`.

### 2. Triage

Run `/triage` (loads `~/.claude/skills/triage`). The skill knows the
canonical role names; the labels above are the local mapping.

Outputs of triage:

- `ready-for-agent` — post an **agent brief** comment on the issue (see
  [the triage skill's AGENT-BRIEF.md](../../../../.claude/skills/triage/AGENT-BRIEF.md)
  in your home install). The brief is the contract the AFK agent works
  from; the issue body is context.
- `ready-for-human` — same shape as an agent brief, but note why a human
  has to do it (judgment, external access, manual testing).
- `needs-info` — comment using the triage skill's needs-info template.
- `wontfix` (enhancement) — write to `.out-of-scope/<concept>.md`,
  link from a closing comment.
- `wontfix` (bug) — close with explanation.

### 3. Implementation

For `ready-for-agent` issues: an AFK agent picks the work up, branches,
implements, opens a PR. The agent brief (not the issue body) is the spec.

For `ready-for-human`: maintainer or human contributor handles directly.

### 4. Vertical slicing

For large features, run **`/to-issues`** on the PRD issue to break it
into independently-grabbable vertical slices. Each slice gets its own
`ready-for-agent` issue.

## AI-generated comments

Per the triage skill, every comment or issue posted by an AI during
triage **must** start with this disclaimer:

```
> *This was generated by AI during triage.*
```

PRD bodies created by `/to-prd` carry the same disclaimer.

## Domain glossary

When writing PRDs, agent briefs, or issue comments, use the canonical
domain vocabulary from [CONTEXT.md](../../CONTEXT.md). Don't invent new
terms. If a term needs sharpening, do it in CONTEXT.md, then reference
it from the issue.

## ADR cross-references

Decisions affecting an area should reference the relevant ADR in
`docs/adr/`. Agent briefs in particular should call out applicable ADRs
so the implementing agent doesn't relitigate settled decisions.
